关于委托(Delegation)的个人理解

Posted on 2018-06-16 | In Java | reading

对于编程思想中很常见的委托(Delegation)之前虽然大概理解，但对于一些细节上的区别还是有些模糊，今天复习时恰好看到，将一些自己的简单理解记录下来，以下内容有自己的理解，也有网上查阅资料后的结果。
委派这个概念其实平时写代码时大都有用到，只是没有刻意去理解，这个概念可以细分为：

  1. 临时性委派(Dependency)
    这是最简单的一种委派方式，调用类里的方法(use a),即在一个类中使用另一个类而不实际将他作为自身属性
  2. 永久性委派（Association）
    类之中有其它类的具体实例来作为一个变量（has a）
  3. 组合(Composition)
    这是更强的委托类型，将一些简单的对象组合为一个更复杂的对象(is part of)
  4. 聚合(Aggregation)
    这种类型可以看作has a，聚合将外部传入的参数对象作为自身的一部分，和组合相似而又有一些区别

其中组合和聚合这两种类型之前我一直没有彻底区分开，明白后发现其实还是很简单的
若谈这两种类型的强弱，组合应该是强于聚合的，在类图中组合是以实心菱形表示，而聚合以空心表示。


组合表示的是(part-of)，也就是说，组合中的组件脱离了整体是没有意义的，例如上第三点的例图中，菜单Menu脱离了窗口window是没有意义的，他没有独立存在的必要，所以在组合中组件是在内部创建的。
而聚合不同，聚合内部的对象是可以脱离整体存在的。举例来说，一个大学中可以有很多个部门，而部门中有很多教授，如果大学不存在了，那么部门显然也无法独立存在。但教授们即使大学不存在也依旧可以独立存在。这可以理解为组合和聚合的区别。
简单来说，组合和聚合的组建对象生命周期不同，组合的类一旦整体对象被回收，则内部组件也就不存在了。但聚合不同，聚合的组件可以独立于整体。
其实可以将组合看作更强的聚合，在实际设计中，应该稍微慎重一点选择使用组合还是聚合。
以上是我个人对于这个问题的一点理解，如有错误，欢迎指正。
